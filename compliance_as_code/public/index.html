Terraform commands

  • terraform fmt : Format our code, or -check that it’s formatted
  • terraform init : Initilize working directory, download providers & modules, and reference state with -backend-config
  • terraform validate : Validates the configuration files. Is the code at least sane?
  • terraform apply -refresh-only : Updates state from remote objects, normally part of terraform plan and terraform apply
  • terraform plan : Outputs an execution plan of the steps needed to apply your code changes to your infrastructure, -out to a file
  • terraform apply : Apply the changes to infrastructure. Can use a plan file

Terraform Process

  • Initialize our code to determine providers to use, download modules, and reference state
  • Plan determines changes to make to provider infrastructure, to match code, referencing state
  • Apply make the planned changes

Feature: Tagged resources

  Scenario Outline: Ensure that specific tags are defined
    Given I have resource that supports tags defined
    When it has tags
    Then it must contain <tags>
    And its value must match the "<value>" regex

    Examples:
      | tags               | value                                          |
      | ApplicationName    | .+                                             |
      | CostCenter         | ^([0-9]{5})$                                   |
      | DataClassification | ^(public\|internal\|confidential\|restricted)$ |
      | Environment        | ^(production\|stage\|development)$             |
      | OwnerEmail         | ^.+@(yibble\.org\|example\.com)$               |

Testing: terratest

  • End-to-end (e2e) testing, because:
    • A valid terraform plan does not mean you’ll get a successful terraform apply
    • outputs from your module may not match expectations
package test

import (
	"testing"
	"fmt"

	"github.com/gruntwork-io/terratest/modules/terraform"
	"github.com/gruntwork-io/terratest/modules/random"
	test_structure "github.com/gruntwork-io/terratest/modules/test-structure"
	"github.com/stretchr/testify/assert"
)

func TestTerraformTemplateExample(t *testing.T) {
	t.Parallel()

	fixtureFolder := "../example"
	uniqueId := random.UniqueId()
	exampleVariable := fmt.Sprintf("example-value-%s", uniqueId)
	
    // Use Terratest to deploy the infrastructure
    test_structure.RunTestStage(t, "setup", func() {
        terraformOptions := &terraform.Options{

			// Set the terraform binary name that the code will be tested with.
			TerraformBinary: "terraform",

			// Indicate the directory that contains the Terraform configuration to deploy
			TerraformDir: fixtureFolder,

			// Vars or Var Files can be used to namespace testing, so testing doesn't impact previously
			// deployed resources:
			Vars: map[string]interface{} {
				"example_variable": exampleVariable,
			},
        }

        // Save options for later test stages
        test_structure.SaveTerraformOptions(t, fixtureFolder, terraformOptions)

        // Triggers the terraform init and terraform apply command
        terraform.InitAndApply(t, terraformOptions)
    })

    test_structure.RunTestStage(t, "validate", func() {

        // run validation checks here
        terraformOptions := test_structure.LoadTerraformOptions(t, fixtureFolder)
		    nullResourceId := terraform.Output(t, terraformOptions, "null_resource_id")
			assert.Regexp(t, "^[0-9]{19}$", nullResourceId)
    })


    // When the test is completed, teardown the infrastructure by calling terraform destroy
    test_structure.RunTestStage(t, "teardown", func() {
        terraformOptions := test_structure.LoadTerraformOptions(t, fixtureFolder)
        terraform.Destroy(t, terraformOptions)
    })
}
